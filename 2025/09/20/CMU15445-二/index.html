CMU15445(二)--存储层

数据的存储：

系统的本质是协调数据在磁盘与内存之间的移动。数据的存储结构总是层状的：

(CPU Register –> CPU Caches(cpu缓存) —> DRAM(内存) —> SSD(机械硬盘) —> HDD —-> Network Storage)

在不同的层次上决定了使用什么数据结构与算法。

在DRAM之前的结构都被称为易失性结构( 拔掉电源后，数据立刻消失)；之后的结构都被称为非易失性结构。

易失性与非易失性结构的区别：

  • 易失性结构都是通过字节寻址的，可以随意转移到目标数值；
  • 非易失性结构都是块状寻址：不能访问任意偏移地址，只能以块为调位调入整个页面。比如：一次性调入4000字节，以来检索到包括的目标数值。

PS: 这里我们统一将非易失性结构统称为磁盘。

数据的访问方式：

顺序访问VS随机访问

  • 随机访问：不断移动到不同的位置，读取目标的数值，访问任意位置的离散的数据。
  • 顺序访问：给定目标数值的附近区域，一次性提取出连续的正片区域的数值。访问连续位置的块数据。

访问速度比较：

由于物理上的移动速度很慢，访问不同物理位置的开销更大，故随机访问的速度要显著慢于顺序访问。

面向磁盘的数据库系统：

基本架构：

存储层：

所有数据都持久化在磁盘上。

数据库在磁盘上通常仅表示为1个或2个文件。但磁盘内部又会将文件分成一个个固定大小的块，称为页。

将数据按照固定的偏移量划分，以便于访问时更快的移动。

缓冲层：

在内存中会存储有缓冲池。是由DBMS管理的内存，用来将数据从磁盘转移到内存中。

被划分为“帧”的数组，用来对应存储磁盘中的“页”。

是存储层的副本，避免每一次操作都要对磁盘进行开销巨大的I/O

数据库的页：

简述：

一个页是一个有着固定大小的块数据。包括了数据的元组，id，记录等信息；

每一个页都会被给予一个独一无二的索引id，用来唯一标识数据

页的id为我们建立了一个对于物理位置的简介映射。

DBMS对页的管理方式：

  • 堆文件；
  • 树文件；
  • 顺序文件；
  • 哈希文件；
  • 索引组织表；

堆文件：

堆文件是一个以随机顺序存储的无序页的集合。

对于定位位置的元数据的维护：
  • 若只有一个文件：

因为page的大小是固定的，只需要页码 * size即可得到偏移量；

  • 若有多个文件：

会维护一个目录页。

目录页会为每一个页保存一个记录，包括：页ID，该页的空闲空间总量。其他的元数据。

页面目录本质可以理解为一个哈希表。

页头：

页面中总会有有个页头起相似的作用，来存放该页的各种元数据

页面中数据的存储方式：

元组的存储：

使用槽页的结构(主要面向行存储方式)

在页头位置会有一个槽数组，固定长度，全部指向页面内的位置。在页尾位置，是所有的固定长度与可变长度的元组数据。

当添加新元组时，会将其放至页脚末尾，元组数据从尾到头拓展，槽数组从头到尾延伸。

当空间耗尽时，他们会在中间相遇。

槽数组存放着page的偏移量：

  • 删除数据：

仅在对应的槽数组中标记特殊值(如-1)，标记该位置无效。

DBMS会通过记录ID来标记每个逻辑元组的物理位置：

1，File ID， pageID， Slot量；

2，大多数DBMS并不会将记录ID存储在元组中。

  • 更新数据：

遵循“先删除后插入”的原则：

先对原有数据进行删除标记，再对余下的page空间进行分析，如果不足够在后面新增记录，就新开一个page，重新设立记录的id。

目前存在的问题：

1，页面空间使用的碎片化：

槽数组与原数组并不总是会正好对齐，中间总会有空余。

2，无意义的对于磁盘的I/O操作：

提取数据的颗粒度是页面大小，即使只需要更新一个元组，我也需要提取整个页面。

3，记录随机分布时的问题：

若需要更新10条记录，每条记录都位于不同的page上，就意味着需要提取10个页面的信息，且大部分都是冗余的。

4，若文件存入磁盘中，我们无法对其进行更新操作，只能对其进行追加。例如各种分布式文件系统。

故采用一种新形式替代元组存储形式：

即日志结构存储：

日志结构存储：

核心思想：

不再使用原地更新的槽页，而是使用一个数据库维护的 日志来记录对元组的所有更改每条日志条目即是代表一个操作。

工作流程：

当我们想要插入一条记录时，首先填充到名为MemTable的表中：

  • 【MemTable表的数据结构是可变的：链表，B+树，B树…..】
  • MemTable表是位于内存中的，所以对其的操作速度很快。
  • MemTable表中的叶节点才是真正存储数据的地方。
  • 当存在相同的键时，用最新的键值去更新原有的。
  • 当MemTable表存满了之后，会沿着叶子结点遍历排序，生成存有条目信息的SSTable表。

SSTable即是日志条目，会记录下内存表中所有更新的内容。

存入SSTable中后，Memtable表中的内容即会释放。

然后会将SSTable写入磁盘。

SSTable在磁盘中的存放方式常常采用分层的方法，基于B+树的形式：

当上一层的存满了之后，会对与排序后的后临近的进行合并，找到SSTable中不同键的最新条目，合并成一个更大的SSTable

优势：

写入速度非常快：

1，是在内存中进行的；

2，不必检查剩余空间，再考虑是否要新开一个page；

3，只需要盲目存入即可，此结构我们只关心最新值。

缺点：

读取速度很慢；因为朴素下是要一页页的检查搜索。

假如我们现在要读取key = 101的值：

若不在就需要向下检索：

在每个层级中进行二分搜索，非常耗时……

其中的偏移数组像槽数组一样进行存储。

数据库的主要工作负载：

主要分为以下几大类：

  • 在线事务处理能力(OLTP)：

    • 主要特征：操作数据量小，但频率极高。
    • 核心目标：快速处理大量用户发起的单个事务，保证数据的一致性和完整性。

  • 在线分析处理(OLAP):

    • 主要特征：一次扫描大量的记录，最终返回汇总的结果。
    • 核心目标：快速扫描和整合海量数据。

  • 混合事务分析处理(HTAP):

    旨在统一数据库中同时处理OLTP和OLAP工作负载。

想要具体选择使用哪一个方式，需要综合考虑READ-Heavy与WRITE-Heavy的综合比重大小。这里采用四象限图的方式展现：

存储模型：

决定了数据在磁盘上如何进行存储，索引与检索。不同的存储模型是为了适应不同的工作负载。

主要有三种模式：行式存储(NSM); 列式存储(DSM); 混合存储类型(PAX);

N元组存储模型(NSM)：

即是将一个记录的N个属性都连续存储在同一个页面。也就是行式存储。

  • 优势：适用于OLTP。访问的元组数量很少，但是对于每一个元组的每一个属性基本都会用到。采用行式存储，避免了不断跳跃，直接连续读取，就可获得例如name = xml 的全部信息。

  • 工作流程：

写入：定位找到空闲页面；连续写入元组数据；更新页头，添加新索引条目，指向对应的行；

  • 缺点：

低效的OLAP查询：若我只想要计算平均年龄，但我又不得不读取地址，姓名，等无关信息，造成了大量的I/O浪费;

数据压缩率较低：同一条记录中不同列的数据类型可能会差异很大，不利于进行数据压缩。

分解存储模型(DSM):

将一个属性的所有元组的值都连续存放在同一页。即列式存储。

整个列文件的开头会存在一个Header，通常包含校验和，数据统计信息，索引指针，数据类型等……

  • DSM的标识符：

    由于列是分开进行存储的，那么需要一个机制来重组完整的行。

    • 依靠固定长度的偏移量：

      每一个元组的值都是按照一样的顺序，占据相同大小的偏移量进行存储的；

      对于长度不固定的VARCHAR文件，blob文件，采用的做法是设计一个字典码的映射，不直接存储文件本身，而是存储映射码。

    • 分配元组id：

      为每一行分配一个唯一的行ID，并在每一个列文件中存储（行ID，列值)

  • 优点：

更快的I/O操作，因为只会读取它所需要的数据；

更利于进行数据压缩；

  • 缺点：

点查询和行重组开销大

不适合小批量的单点查询。

PAX模型：

基本思想：既想要获得列存储的优势，又想要保留空间局部性，即单个表的数据彼此接近。

实现流程：

现将元组水平划分为一个个行组，再在每个行组内竖直划分为列组。

数据的压缩：

读取文件速度的瓶颈即是I/O操作，现在列式存储已经实现了最少操作次数，那么提高每一次读取时的文件数量，便是进一步优化的关键

压缩的问题：

1，必须要是无损压缩：解压后的文件必须与原文件保持一致。

2，延迟实例化：不能为了查询一条记录，而要解压例如十亿条数据。

压缩的形式：

块级压缩：

数据库将一个数据页中的所有行为作为一个整体进行压缩。

常用算法：LZ77/LZSS; LZ4; Zstd（最佳）；

工作流程：

当一页被填满，调用压缩算法对其进行压缩；

压缩后的页被写入磁盘；

需要读取时，将其写入内存并解压。

缺点：

需要读取或更新任何一条记录时，都需要对整个页面进行解压，增加了开销。

mysql对于块级压缩的实现：

mysql对于每一个压缩快采取了日志结构的存储。mod log页头存储了对应的改变，且是未进行压缩的。

在进行更新时，更新数据会直接存进mod log中，而不会对页内容进行解压(除非指定某一页的某条记录进行何种改变)。

在进行读取时，会首先查看mod log中是否会有想要查询的内容。只有没有时，才会将file文件解压合并到内存中。

位图编码：

基本思想：不是在列中重复存储值，而是为每一个可能出现的值维护一个单独的位图。

故适用于低基数的情况，即答案可能性很少。

数据库管理系统（DBMS）：DBMS 是一种允许应用程序在数据库中存储和分析信息的软件。 DBMS 允许根据某些数据模型来定义，创建，查询，更新，管理数据库。 数据模型：是一种描述数据库中数据的概念的集合： 实例：关系，NoSQL(键/值，文档，图形)，数组/矩阵/向量(用于机器学习) 关系数据模型：关系数据模型定义了基于关系的数据库抽象概念，以避免更新数据库而要重写代码的开销。 关系模型三个原则：1，用简单的数据结构来存储数据； 2，物理存储方式由 DBMS 的具体实现来决定。 3，只通过 sql 语言接受数据，DBMS 找出来最佳的执行策略。 关系模型定义三个概念：1，结构：关系的定义及其内容。关系具有的属性，以及这些属性可以包含的值 2，完整性：确保数据库的内容满足约束。如：year 属性的值必须均为数字 3，操作：如何访问和修改数据库的内容 元组是关系中的一组属性值。每个属性都可以有特殊值 NULL，这表示该属性是未定义的。 具有 n 个属性的关系被称为 n 元关系。n 元关系相当于具有 n 列的表。 key:主键：一个关系的主键独一无二地标...
